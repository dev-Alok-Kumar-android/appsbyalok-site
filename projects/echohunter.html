Technical Difficulties Solved

Real engineering challenges encountered while building a native game engine from scratch.

Zero-GC Game Loop

Problem: Standard object instantiation inside the Android onDraw() loop triggered constant GC pauses, ruining the 60fps experience.
Solution: Eliminated all object creation in the game loop. Utilized strictly pre-allocated primitive arrays (FloatArray, IntArray) via an Object Pooling pattern.

Camera & Rotation Sync

Problem: Camera offsets, collision checks, touch controls, and renderers can desync when the world is larger than the screen or the device rotates mid-run.
Solution: Centralized world coordinates in GameState, kept camera math synced across systems, and added rotation guards to pause unstable transitions cleanly.

Procedural Mission Variety

Problem: Random maps quickly become repetitive or unfair when every level uses the same win condition and enemy pressure.
Solution: Built a LevelEngine feature matrix that combines mission types with MazeGenerator layouts, reward scaling, enemy speed, HP, spawn rate, and behavior intelligence.

Physics & Tile Collision

Problem: Without Box2D, player movement against procedural maze walls felt sticky or glitchy.
Solution: Developed a custom Axis-Aligned Bounding Box (AABB) tile-sliding collision system, calculating vector penetration depth to ensure buttery-smooth corner sliding.

Enemy Behavior Scaling

Problem: Enemies need to feel smarter over time without becoming impossible or ignoring walls, gates, core objectives, and boss arenas.
Solution: Tuned intelligence from early simple behavior toward expert pressure, with separate systems for enemy updates, boss state, player assist logic, and objective-aware progression checks.

Persistent Roguelite State

Problem: Story streaks, hard-mode unlocks, archives, auto-next, and upgrade purchases must survive app restarts without corrupting active runs.
Solution: SaveManager stores campaign progress, unlocked streaks, current streaks, and preferences separately from GameState's live run snapshot.

Development Philosophy

Echo Hunter was born out of a desire to understand what happens "under the hood" of massive game engines like Unity or Godot.

Instead of relying on drag-and-drop interfaces and heavy binary bloat, the goal was to manually code the physics math, the render loops, and the enemy pathfinding algorithms using pure, low-level Android APIs.

The result is an experiment in extreme optimization. By strictly controlling memory allocation and leveraging native hardware drawing via the Canvas API, the project keeps tactical gameplay, procedural missions, and a terminal UI inside a compact native Android build.
