Technical Difficulties Solved

Real engineering challenges encountered while building a native game engine from scratch.

Garbage Collection Stutters

Problem: Standard object instantiation inside the Android onDraw() loop triggered constant GC pauses, ruining the 60fps experience.
Solution: Eliminated all object creation in the game loop. Utilized strictly pre-allocated primitive arrays (FloatArray, IntArray) via an Object Pooling pattern.

Screen Rotation State Loss

Problem: Device rotation recalculates surface dimensions, breaking custom physics coordinates and UI layouts.
Solution: Implemented a dynamic layout math system that rescales hitboxes on the fly, paired with a strict engine pause state to prevent out-of-bounds deaths during rotation.

Complex AI in Pure Logic

Problem: Handling pathfinding for dozens of enemies in procedural mazes without a pre-built navigation mesh.
Solution: Hand-coded a Breadth-First Search (BFS) Heatmap generator and Line-of-Sight (LoS) Raycasting algorithms that execute entirely on background thread pools to keep UI rendering smooth.

Physics & Tile Collision

Problem: Without Box2D, player movement against procedural maze walls felt sticky or glitchy.
Solution: Developed a custom Axis-Aligned Bounding Box (AABB) tile-sliding collision system, calculating vector penetration depth to ensure buttery-smooth corner sliding.

Development Philosophy

Echo Hunter was born out of a desire to understand what happens "under the hood" of massive game engines like Unity or Godot.

Instead of relying on drag-and-drop interfaces and heavy binary bloat, the goal was to manually code the physics math, the render loops, and the AI pathfinding algorithms using pure, low-level Android APIs.

The result is an experiment in extreme optimization. By strictly controlling memory allocation and leveraging native hardware drawing via the Canvas API, the project achieves a fully-featured, tactical gaming experience in under 150 Kilobytes.
